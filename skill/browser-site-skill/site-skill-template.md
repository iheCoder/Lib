# 站点技能模板

当你成功探索一个网站的自动化流程后，使用本模板将其沉淀为可复用的**站点技能文档**。

## 为什么需要站点技能

- **避免重复探索**：同一个网站的流程，不需要每次都从零开始理解
- **知识沉淀**：把页面结构、关键目标、状态机、异常分支等隐性知识显性化
- **参数化复用**：抽象出变化项（如搜索词、日期、筛选条件），保留稳定部分
- **团队共享**：让其他人或 Agent 也能快速上手这个网站

## 何时创建站点技能

### 适合场景 ✅

- 同一个网站的流程需要多次执行（如每日爬取、定期检查、批量操作）
- 网站结构相对稳定，关键目标不经常变化
- 流程有明确的输入参数和成功判定标准
- 需要团队共享或长期维护

### 不适合场景 ❌

- 一次性任务，不会再执行
- 网站结构极不稳定，每周都改版
- 流程过于简单，只有 1-2 步操作
- 没有明确的成功判定标准

## 工作流步骤

### 流程 1: [流程名称，如商品搜索与加购]

**输入参数**
- `search_keyword`: 搜索关键词（必填）
- `price_range`: 价格范围（可选，默认不限）
- `quantity`: 购买数量（可选，默认 1）

**执行步骤**
1. **打开首页**
   - 访问 https://example.com
   - 验证：页面标题包含"[网站名称]"

2. **搜索商品**
   - 使用目标：`product_search_input`
   - 填写：`search_keyword`
   - 验证：搜索结果列表出现

3. **筛选价格**（如果 `price_range` 提供）
   - 使用目标：`price_filter`
   - 选择：`price_range`
   - 验证：结果列表更新

4. **选择第一个商品**
   - 使用目标：`first_product_link`
   - 点击
   - 验证：进入商品详情页

5. **加入购物车**
   - 使用目标：`add_to_cart_button`
   - 点击
   - 验证：购物车数量 +1 或加购成功提示

**成功判定**
- 购物车中包含指定商品
- 商品数量与 `quantity` 一致
- 无错误提示

**失败场景**
- 搜索无结果：记录日志，返回"未找到商品"
- 商品缺货：跳过该商品，尝试下一个
- 加购失败：重试 2 次，仍失败则报告

## 决策点

在自动化过程中需要做出决策的关键点：

### 决策点 1: 是否需要登录

**判断依据**
- URL 是否被重定向到登录页
- 页面是否显示"请登录"提示
- 是否存在用户头像元素

**决策逻辑**
- 如果需要登录 → 执行登录流程
- 如果已登录 → 继续原流程
- 如果无法判断 → 尝试访问需登录的页面，根据响应决定

---

### 决策点 2: 是否有验证码

**判断依据**
- 页面是否包含 captcha/验证码元素
- 登录/提交按钮是否被禁用

**决策逻辑**
- 如果有验证码 → 暂停并请求人工介入
- 如果无验证码 → 继续自动化流程

## 成功判定

### 整体成功标准

满足以下所有条件视为流程成功：
- [ ] 所有关键步骤都已执行
- [ ] 所有 `state_after` 验证都通过
- [ ] 无报错或异常提示
- [ ] 最终状态符合预期（如购物车有商品、订单已提交）

### 部分成功标准

某些场景允许部分成功：
- 批量操作：至少 80% 的项目成功
- 数据抓取：至少获取到 50% 的目标数据
- ...

## 异常分支

### 异常 1: 网络超时

**触发条件**
- 页面加载超过 30 秒
- 请求无响应

**处理策略**
1. 刷新页面
2. 如果仍超时，等待 1 分钟后重试
3. 重试 3 次后报告失败

---

### 异常 2: 元素未找到

**触发条件**
- 多模态匹配失败
- fallback selector 也失败

**处理策略**
1. 截图保存当前页面
2. 检查页面结构是否改变
3. 报告"站点技能需要更新"

---

### 异常 3: 意外弹窗

**触发条件**
- 出现广告弹窗
- 出现 Cookie 同意弹窗
- 出现活动推广弹窗

**处理策略**
1. 识别弹窗类型
2. 尝试点击"关闭"或"拒绝"按钮
3. 如果无法关闭，跳过该步骤

## 漂移信号

当出现以下信号时，表示网站可能已改版，站点技能需要更新：

- ⚠️ 关键目标连续 3 次匹配失败
- ⚠️ DOM 路径摘要不再匹配
- ⚠️ 页面布局明显变化（通过截图对比）
- ⚠️ 新增了之前没有的验证流程（如验证码、人机验证）
- ⚠️ 流程步骤顺序改变（如登录前置、新增中间步骤）

**更新建议**
- 重新探索该网站
- 更新所有失效的多模态描述
- 补充新增的决策点和异常分支
- 更新"最后验证"日期

## 禁止操作

为避免风险，以下操作严禁执行：

- ❌ **禁止提交真实订单**（除非用户明确授权）
- ❌ 禁止删除用户数据
- ❌ 禁止修改账户设置
- ❌ 禁止点击"支付"或"扣款"相关按钮
- ❌ 禁止发送消息、评论、发帖（除非用户明确授权）
- ❌ ...

**安全建议**
- 在测试环境或沙箱账户中验证
- 对危险操作设置二次确认
- 记录所有操作日志，便于审计

# 使用本模板的说明

## 填写指南

1. **不要一次性填完所有字段**：根据实际需要逐步补充
2. **从最小集合开始**：先填写站点概览、入口、关键目标、核心流程
3. **逐步补充**：在后续使用中遇到问题时，再补充决策点、异常分支等
4. **保持更新**：网站改版后及时更新，标记状态和维护记录

## 优先级建议

### 第一版（最小可用）
- ✅ 站点概览
- ✅ 入口与导航
- ✅ 关键目标定义（核心 3-5 个目标）
- ✅ 工作流步骤（1 个核心流程）
- ✅ 成功判定

### 第二版（增强稳定性）
- ✅ 异常分支（2-3 个常见异常）
- ✅ 决策点（1-2 个关键决策）
- ✅ 漂移信号

### 第三版（完善维护）
- ✅ 状态机
- ✅ 禁止操作
- ✅ 依赖与前置条件
- ✅ 维护记录

## 文件组织建议

### 单文件方式
适合简单站点，把所有内容放在一个 `.md` 文件中。

### 多文件方式
适合复杂站点，按功能拆分：

```
site-skills/
├── example-com/
│   ├── README.md              # 站点概览
│   ├── targets.md             # 关键目标定义
│   ├── workflows/
│   │   ├── search-and-buy.md # 工作流 1
│   │   ├── order-query.md    # 工作流 2
│   │   └── ...
│   ├── exceptions.md          # 异常分支
│   └── maintenance.md         # 维护记录
```

## 与 open-prose 集成

如果你使用 OpenProse，可以把站点技能转换为 `.prose` 程序：

```prose
# example-com-search.prose

input search_keyword: "搜索关键词"
input price_range: "价格范围（可选）"

# 加载站点技能
use "site-skills/example-com.md" as site_skill

session search:
  prompt: """
    在 example.com 搜索商品：{search_keyword}
    使用站点技能中定义的多模态目标描述
  """
  context: { site_skill, search_keyword }

session filter:
  prompt: "应用价格筛选：{price_range}"
  context: { site_skill, price_range }

session add_to_cart:
  prompt: "将第一个商品加入购物车"
  context: site_skill

output result = {
  status: "success",
  product: "...",
  price: "..."
}
```
