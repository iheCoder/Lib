# Shadow Claw — 影爪调查引擎

> **不是"帮你看看日志"，而是像高级工程师一样驱动一轮完整的事故调查。**

## 核心哲学

Shadow Claw 的价值不是猜得准，而是 **知道自己哪里不确定，并主动去消除不确定性**。

> 没有 Validator，系统会在一个错误世界里越分析越自信。
> 有 Validator，它会不断逼你跳出错误世界。

### 铁律：理解先于修复

> **在调查收敛之前，禁止修改任何代码、配置或基础设施。**
>
> 改代码是处置，不是调查。处置在 Phase 6，必须在理解问题之后。
> 你可能在修一个根本不存在的 bug。

---

## 世界意识（World Consciousness）

**这是整个 Shadow Claw 最核心的概念。理解了它，其他一切都是自然推论。**

一个故障的真相可能藏在任何"世界"里——代码、日志、配置、部署、网络、数据库、基础设施……
你能看到哪些世界，决定了你能提出哪些假设。
**你看不见的世界，才是真正危险的地方——因为你甚至不会为它生成假设。**

### 调查开始前，先画世界地图

在做任何事之前，花 30 秒想清楚：

```
这个症状可能涉及哪些世界？
  ✅ 能进入     — 我有工具/权限，可以采集证据
  ⚠️ 部分可见   — 能看到一些，但不完整
  ❌ 进不去     — 没有工具/权限，这是我的盲区
```

这张地图的意义：

> **你的置信度上限 = 你能看到的世界的覆盖率。**
> 如果 3 个强相关世界里有 2 个是盲区，你不可能给出高置信度结论。

### 世界意识如何驱动调查

这不是一个额外的步骤，这就是调查循环本身的运转方式：

```
1. 在可见世界中采集证据、提出假设
2. 用 Validator 审判——如果可见世界的证据解释不了症状
3. 自然地把目光转向其他世界——"答案可能在我看不到的地方"
4. 能进入 → 扩展世界，采集新证据
5. 进不去 → 坦诚告诉用户：我的盲区在哪里，我需要什么
```

**承认无知不是失败，是调查在正确运转的标志。**

## 定位

Shadow Claw 是调查系统的 **编排器和循环驱动器**。

它不亲自做事实提取（那是 Evidence Builder 的事），
也不亲自做假设审判（那是 Hypothesis Validator 的事），
它做的是 **驱动循环，直到收敛或诚实地标注无知边界**。

### 三个角色，一个循环

```
  Evidence Builder  →  Hypothesis Engine  →  Hypothesis Validator
      (事实层)             (猜测层)               (审判层)
        ^                                            |
        |         解释不成立 → 扩展世界 或 承认无知     |
        +--------------------------------------------+
```

| 角色 | 决定什么 |
|------|---------|
| Evidence Builder | 你 **看到了什么** |
| Hypothesis Engine | 你 **猜是什么** |
| Hypothesis Validator | 你 **该去看什么**（或该承认看不见什么） |

> **Evidence Builder 不会主动扩展世界。是 Hypothesis Validator 在"逼它去扩展世界"。**

### Capability Pool（能力池）

进入不同"世界"需要不同的工具。大部分是天然能力（`grep`、`git`、`kubectl`），
有些需要专门 skill：

| 外部能力 | 能进入的世界 | 说明 |
|---------|------------|------|
| aliyun-sls-trace | trace, log | 阿里云链路追踪与日志查询 |
| mysql-readonly-query | database | MySQL 只读查询 |
| web-access | external_knowledge, deploy, network, environment | 外部信息获取、部署平台访问 |

**当你发现某个世界进不去时**，先查能力池有没有对应工具。
如果真的没有——这就是你该向用户坦诚的时刻。

详见 [references/capability-pool.md](references/capability-pool.md)。

## 何时使用

- 用户报告系统故障、性能异常、请求超时等
- 需要从症状出发，找到根因或至少确定问题方向
- 问题复杂到不能一眼看出答案（多源、跨层、表象模糊）

## 调查循环

### 总览

```
症状 → 画世界地图 → 循环 { 采集 → 假设 → 验证 → [扩展世界 | 承认无知] } → 报告 → 处置
```

最多 **5 轮**。每轮世界变大，认知升级。随时可以向用户求助——不要闭门造车。

---

### Phase 0: 接收症状 (Intake)

1. 从症状中提取: 实体、时间窗、环境、症状描述
2. **画世界地图**：列出所有可能相关的世界，标注 ✅⚠️❌
3. **向用户确认**：提取是否准确？有没有遗漏的上下文？
4. 检查 project-cognition 中是否有相关历史经验

---

### Phase 1: 初始观测 (Observe)

**调用 Evidence Builder** 在可见世界中采集证据。

**关键**: 不仅采集"有什么"，还要主动检查"没有什么"（缺失事实）。
不可达的世界记录为缺失事实，影响置信度上限。

---

### Phase 2: 假设生成 (Hypothesize)

基于当前证据包，生成候选假设。

- 不能只有一个假设——至少要有竞争假设来保持诚实
- 假设应覆盖不同层级（不要全猜在同一层）
- 每个假设必须能被证伪
- **为盲区世界也生成假设**——即使你无法验证，它们的存在提醒你置信度有天花板

层级参考: 代码层 / 配置层 / 部署层 / 依赖层 / 基础设施层 / 观测层

---

### Phase 3: 假设验证 (Validate)

**调用 Hypothesis Validator** 做四层校验（时间 → 量级 → 范围 → 反事实）。

```bash
python3 hypothesis_validator/scripts/validate_hypothesis.py --input evidence.json --pretty
```

然后由 LLM 补充反事实校验。

---

### Phase 4: 决策 (Decide)

- **有假设被 promote** → 收敛，Phase 5
- **证据不足 (insufficient_evidence)** → 扩展世界，回 Phase 1
- **有 retain 但无法区分** → 扩展世界
- **全部 reject** → 生成新假设或带不确定性收敛
- **到达最大轮次** → 带不确定性收敛

#### 扩展世界

按**判别性信息增益**排序（见 investigation-loop-protocol.md），
选最能区分竞争假设的世界扩展。新证据增量追加到 Evidence Pack。

#### 当世界进不去时

**这是最关键的分支。不要静默跳过，不要硬在已有世界里凑答案。**

向用户坦诚：

> "我已排查了 [code/log/trace] 世界的证据，但不足以解释症状。
> 我怀疑问题可能在 [deploy/network/infra] 层面，但我无法进入这些世界。
> 你能帮我确认：[具体问题]？"

**一个诚实的"我不知道"胜过一个自信的错误结论。**

---

### Phase 5: 收敛与报告 (Converge)

报告是给人看的，用 markdown。结构：

1. **一句话结论**（或一句话标注不确定性边界）
2. **关键证据链**——按时间排列的因果链
3. **被推翻的假设**——推翻理由是调查严谨性的证明
4. **剩余不确定性**——哪些世界没进入？哪些假设无法验证？
5. **调查过程摘要**

---

### Phase 6: 处置 (Resolve)

处置方案分四级: 🔴立即止血 → 🟡短期修复 → 🔵长期预防 → 📊持续监控

执行原则——**按副作用分级**:

| 副作用 | 处理方式 |
|--------|---------|
| 无（只读/分析/创建分支） | 直接执行 |
| 低（本地配置/新分支提交） | 直接执行 |
| 中（扩缩容/限流） | 征询用户后执行 |
| 高（回滚/重启/生产变更） | 征询用户后执行，或只给方案 |

---

## 用户交互

调查不是闭门造车。**但只在真正需要时才问用户**——能自己查到的不要问。

核心交互时机:

| 时机 | 做什么 |
|------|--------|
| Phase 0 | 确认症状提取、补充上下文 |
| 可见世界的假设解释不了症状时 | 坦诚盲区，请求帮助进入新世界 |
| 处置方案生成后 | 确认方案、授权有副作用的操作 |

## 反模式：一个真实的失败案例

**症状**：测试环境 `super_ai_update_user_profile context deadline exceeded`

**Agent 做了什么**（全错）：
1. 跳过 Intake，直接钻进代码世界
2. 在本地代码里找到"看起来可疑"的超时处理
3. 跳过假设生成、跳过验证、跳过世界扩展
4. 直接改代码、建文件、跑测试，宣布"已修复"

**真正的根因**：测试环境 DNS 把目标域名解析到一个不可达的内网 IP，Pod 根本连不上目标服务。纯基础设施问题，和代码无关。

**问题出在哪**：不是"应该多加一条检查网络的规则"，而是 **循环根本没运转**。

如果循环正常运转：
1. 画世界地图 → deploy ⚠️、network ❌、infra ❌ 都是盲区
2. 在 log 世界采集证据 → 看到超时，但没有代码错误迹象
3. 生成假设 → H1(代码超时处理) + H2(网络不可达) + H3(DNS问题)
4. 验证 → H1 通不过（代码能解释"处理慢"，解释不了"完全超时"）
5. H2/H3 需要 network 世界但进不去 → **向用户坦诚盲区**
6. 用户帮忙确认 → DNS 解析到内网 IP → 根因锁定

**教训**：不需要特殊规则。**循环本身就是答案——只要它真的在运转。**

## 安全约束

1. **不在证据不足时给唯一结论** — 宁可说"我还不确定，我的盲区在 X"
2. **不跳过假设直接给答案** — 必须有竞争假设
3. **不无限循环** — 最多 5 轮
4. **不忽略 insufficient_evidence** — 必须扩展世界或承认无知
5. **不在调查收敛前改代码** — 铁律
6. **不在盲区里给高置信度结论** — 置信度受世界覆盖率约束

## 与子 Skill 的关系

```
shadow-claw (编排器)
  ├── evidence-builder (事实层)
  │     ├── 初始采集: Phase 1
  │     ├── 扩展采集: Phase 4 (由 Validator 触发)
  │     └── 通过能力池进入各世界
  ├── hypothesis-validator (审判层)
  │     ├── 四层校验: Phase 3
  │     └── 输出 next_worlds_to_query → 驱动世界扩展
  ├── project-cognition (经验层)
  │     ├── 调查前: 读取历史经验
  │     └── 调查后: 沉淀新发现
  └── 用户 (人类层)
        ├── 提供上下文、盲区信息
        ├── 确认处置方案
        └── 授权有副作用的操作
```
