# Shadow Claw — 影爪调查引擎

> **不是"帮你看看日志"，而是像高级工程师一样驱动一轮完整的事故调查。**

## 核心哲学

Shadow Claw 的价值不是猜得准，而是 **知道自己哪里不确定，并主动去消除不确定性**。

> 没有 Validator，系统会在一个错误世界里越分析越自信。
> 有 Validator，它会不断逼你跳出错误世界。

### 铁律：理解先于修复

> **在调查收敛之前，禁止修改任何代码、配置或基础设施。**
>
> 改代码是处置，不是调查。处置在 Phase 6，必须在理解问题之后。
> 你可能在修一个根本不存在的 bug。

---

## 多视角世界模型（Multi-Perspective World Model）

**这是整个 Shadow Claw 最核心的概念。理解了它，其他一切都是自然推论。**

一个故障的真相可能藏在任何"世界"里，而世界不是预定义的分类列表，
而是**从症状出发，通过多个视角推导出来的**。

### 五个必须过一遍的视角

复杂故障可能隐藏在任何维度。每次调查开始时，必须从以下视角构建对系统的理解：

| 视角 | 核心问题 | 典型产出 |
|------|---------|---------|
| **执行路径** | 操作从发起到完成，物理上经过什么？ | 路径上的节点列表，每个节点可能的问题 |
| **时间/变更** | 最近什么变了？发版、配置、依赖、流量？ | 变更事件时间线，与故障的时间相关性 |
| **资源/竞争** | 涉及哪些共享资源？谁可能在竞争？ | 资源-消费者关系，潜在冲突点 |
| **反馈/放大** | 有没有正反馈回路？重试、告警触发、自动扩缩容？ | 放大机制识别，放大因子估算 |
| **数据流** | 数据从哪来？经过什么转换？（如涉及数据问题） | 数据来源和转换链路 |

**不是选择性使用。** 每次调查都快速过一遍这些视角。
有些视角会产出"不相关"——这本身就是有价值的排除。

### 故障族识别

复杂故障通常属于以下经典族之一。**识别故障族有助于确保假设覆盖了正确的方向**：

| 故障族 | 典型信号 | 关键视角 |
|--------|---------|---------|
| **网络/连接** | timeout、connection refused、DNS 失败、TLS 错误 | 执行路径（物理链路每一跳） |
| **配置/变更** | 发版后异常、配置生效后异常、升级后回归 | 时间/变更（什么变了？何时？） |
| **资源/竞争** | CPU/内存飙升、死锁、饥饿、连接池耗尽 | 资源/竞争（谁在抢什么？） |
| **级联/放大** | 部分故障演变成全局故障、重试风暴 | 反馈/放大（放大机制是什么？） |
| **数据/状态** | 数据错误、状态不一致、幂等性问题 | 数据流（数据从哪被污染？） |
| **依赖/第三方** | 间歇性故障、第三方 SLA 波动、库 bug | 依赖边界（版本、兼容性、SLA） |

**故障族不是互斥的。** 一个故障可能涉及多个族（如"配置变更导致网络不可达"）。

### 操作边界定位

对于涉及请求/操作流的故障，回答两个问题来定位方向：

1. **最远确认边界**：操作确认走到了哪一步？（有日志/trace/证据支撑）
2. **第一处未确认边界**：从哪一步开始，没有直接证据？

```
示例：HTTP 请求 timeout
最远确认边界：请求已从 Pod 发出（有客户端日志）
第一处未确认边界：DNS 解析后 → TCP 连接建立前（无证据确认 TCP 握手）

→ 方向：问题可能在 DNS 解析结果（IP 错误）或 IP 可达性（路由/防火墙）
```

操作边界适用于各种操作流：
- **同步 RPC**：Client → LB → App → DB
- **异步 MQ**：Producer → Broker → Consumer
- **网络请求**：Pod → DNS → IP → TCP → HTTP → 上游
- **数据库事务**：Begin → Query → Lock → Commit

**边界定位不需要知道精确原因，但能指明下一步该在哪里找证据。**

### 为什么需要多视角？

不同类型的故障，根因隐藏在不同视角：

- **超时/连接失败** → 执行路径视角最关键（DNS → 路由 → TCP → 应用）
- **发版后故障** → 时间/变更视角最关键（什么变了？）
- **CPU/内存飙升** → 资源/竞争视角最关键（谁在抢资源？）
- **级联雪崩** → 反馈/放大视角最关键（重试风暴？熔断失效？）
- **数据错误** → 数据流视角最关键（数据从哪被污染？）

**如果只用一个视角，你会被这个视角的盲区困住。**

### 从视角产出推导世界

每个视角的产出，就是这次调查需要关注的"世界"：

```
执行路径视角产出：
  [Pod] → [DNS] → [路由] → [网关] → [上游服务]
  
  → 这就是 5 个世界，每个都可能是问题所在
  → 为每个世界标注：✅能进入 / ⚠️部分可见 / ❌进不去
```

```
时间/变更视角产出：
  - 3小时前：上游服务发版
  - 1天前：本服务配置变更
  - 1周前：依赖库升级
  
  → 每个变更事件都是一个假设方向
```

### 世界可达性与置信度

> **你的置信度上限 = 你能看到的世界的覆盖率。**

如果执行路径上有 5 个节点，你只能看到 2 个，你不可能给出高置信度结论。

标注每个世界的可达性：
- ✅ **能进入** — 有工具/权限，可以采集证据
- ⚠️ **部分可见** — 能看到一些，但不完整
- ❌ **进不去** — 没有工具/权限，这是盲区

**进不去的世界也要生成假设** — 它们的存在提醒你置信度有天花板。

### 世界意识如何驱动调查

```
1. 从多个视角构建世界模型
2. 在可见世界中采集证据、提出假设
3. 用 Validator 审判——如果可见世界的证据解释不了症状
4. 自然地把目光转向其他视角、其他世界
5. 能进入 → 扩展世界，采集新证据
6. 进不去 → 坦诚告诉用户：我的盲区在哪里，我需要什么
```

**承认无知不是失败，是调查在正确运转的标志。**

## 定位

Shadow Claw 是调查系统的 **编排器和循环驱动器**。

它不亲自做事实提取（那是 Evidence Builder 的事），
也不亲自做假设审判（那是 Hypothesis Validator 的事），
它做的是 **驱动循环，直到收敛或诚实地标注无知边界**。

### 三个角色，一个循环

```
  Evidence Builder  →  Hypothesis Engine  →  Hypothesis Validator
      (事实层)             (猜测层)               (审判层)
        ^                                            |
        |         解释不成立 → 扩展世界 或 承认无知     |
        +--------------------------------------------+
```

| 角色 | 决定什么 |
|------|---------|
| Evidence Builder | 你 **看到了什么** |
| Hypothesis Engine | 你 **猜是什么** |
| Hypothesis Validator | 你 **该去看什么**（或该承认看不见什么） |

> **Evidence Builder 不会主动扩展世界。是 Hypothesis Validator 在"逼它去扩展世界"。**

### Capability Pool（能力池）

进入不同"世界"需要不同的工具。大部分是天然能力（`grep`、`git`、`kubectl`），
有些需要专门 skill：

| 外部能力 | 能进入的世界 | 说明 |
|---------|------------|------|
| aliyun-sls-trace | trace, log | 阿里云链路追踪与日志查询 |
| mysql-readonly-query | database | MySQL 只读查询 |
| web-access | external_knowledge, deploy, network, environment | 外部信息获取、部署平台访问 |

**当你发现某个世界进不去时**，先查能力池有没有对应工具。
如果真的没有——这就是你该向用户坦诚的时刻。

详见 [references/capability-pool.md](references/capability-pool.md)。

## 何时使用

- 用户报告系统故障、性能异常、请求超时等
- 需要从症状出发，找到根因或至少确定问题方向
- 问题复杂到不能一眼看出答案（多源、跨层、表象模糊）

## 调查模式：渐进式披露

**不是所有问题都需要完整的深度调查。从最轻量开始，必要时升级。**

```
Level 0: 直接回答  ──升级触发──→  Level 1: 快速排查  ──升级触发──→  Level 2: 深度调查
```

### 第一步：强制 Level 判断（必须做）

**收到症状后，首先判断起始 Level，并明确宣告：**

```
【Level 判断】
症状信号：[描述症状的关键特征]
判断依据：[为什么选这个 Level]
→ 进入 Level X
```

| 信号 | 起始 Level |
|------|-----------|
| 错误信息完全明确（文件名、行号、具体错误类型） | Level 0 |
| 单服务内部问题 + 有明确报错 | Level 1 |
| **涉及跨服务调用**（A 调 B 超时/失败） | **Level 2** |
| **涉及网络/连接**（timeout、connection refused、DNS） | **Level 2** |
| 间歇性/分布式/症状模糊 | Level 2 |
| 用户说"很奇怪"/"排查很久了" | Level 2 |

**不确定时，从 Level 1 开始，但跨服务调用问题必须进入 Level 2。**

---

### Level 0: 直接回答

**适用**：错误信息完全明确，答案显而易见

```
症状：FileNotFoundError: /etc/app/config.yaml
→ 直接回答：文件不存在，检查路径、权限、是否漏挂载 ConfigMap
```

**升级触发**：用户说"不对"、"已经检查过了"

---

### Level 1: 快速排查

**适用**：单服务内部问题 + 有明确报错

**流程**：
1. 生成 1-2 个最可能的假设
2. 快速验证（看日志、读代码、查配置）
3. 验证通过 → 给出结论
4. 验证失败 → **升级到 Level 2**

**升级触发**（任一条满足就升级）：
- 快速假设被推翻
- 无法解释症状的量级或范围（如"为什么全部失败而不是部分"）
- 发现涉及多个组件/服务/网络
- 证据指向多个互相矛盾的方向
- 快速验证后问题依然不清晰

---

### Level 2: 深度调查

**适用**：
- Level 1 升级上来
- 症状本身复杂（跨系统、间歇性、分布式）
- 用户明确说"这个问题排查很久了"或"很奇怪"
- 间歇性/分布式/症状模糊

**进入 Level 2 时，必须明确宣告并按结构执行：**

```
【进入 Level 2 深度调查】

一、故障族识别
主要故障族：[网络/连接 | 配置/变更 | 资源/竞争 | 级联/放大 | 数据/状态 | 依赖/第三方]
可能涉及的其他族：[...]

二、操作边界定位
[发起方] → [节点1] → [节点2] → ... → [目标方]
最远确认边界：[哪一步有证据]
第一处未确认边界：[从哪里开始没有证据]

三、世界可达性
✅ 能进入：[...]
⚠️ 部分可见：[...]
❌ 进不去（盲区）：[...]

四、假设列表（必须覆盖多个视角）
- H1: [假设] ← 来自 [视角]
- H2: [假设] ← 来自 [视角]
- H3: [假设] ← 来自 [视角]（盲区假设）
...
```

**Level 2 必须输出的结构化报告：**

```
【Level 2 调查报告】

一、结论（或不确定性边界）
[一句话结论，或"方向是 X，但无法确定具体原因因为 Y 是盲区"]

二、证据链
1. [时间] [事实] ← 来源：[log/trace/...]
2. [时间] [事实] ← 来源：[...]
...

三、假设验证结果
| 假设 | 验证结果 | 理由 |
|------|---------|------|
| H1 | reject/retain/insufficient | [理由] |
| H2 | ... | ... |

四、剩余不确定性
- 盲区：[哪些世界进不去]
- 未验证假设：[哪些假设因为盲区无法验证]

五、需要用户帮助（如有盲区）
- 问题1：[具体问题]
- 问题2：[...]
```

**如果有盲区，必须向用户求助，不要硬在可见世界里凑答案。**

---

### Level 1 升级检查

**在 Level 1 准备给出结论前，强制检查**：

- [ ] 我的假设能**完整解释**症状吗？（量级、范围、时间都对得上）
- [ ] 有没有我**跳过的中间环节**？（操作边界是否连续）
- [ ] 有没有**更简单的解释**被我忽略了？
- [ ] 如果我错了，用户会浪费多少时间？

**任何一项触发，升级到 Level 2 而不是硬给结论。**

## 调查循环（Level 2 深度调查）

**当 Level 0/1 无法解决问题时，进入完整的深度调查流程。**

### 总览

```
症状 → 构建世界模型 → 循环 { 采集 → 假设 → 验证 → [扩展世界 | 承认无知] } → 报告 → 处置
```

最多 **5 轮**。每轮世界变大，认知升级。随时可以向用户求助——不要闭门造车。

---

### Phase 0: 接收症状与构建世界模型 (Intake & World Model)

1. 从症状中提取: 实体、时间窗、环境、症状描述
2. **识别故障操作**：这个症状涉及的核心操作是什么？
3. **多视角构建世界模型**：
   - **执行路径**：这个操作从发起到完成，物理上经过什么节点？
   - **时间/变更**：最近有什么变化？（发版、配置、依赖、流量模式）
   - **资源/竞争**：涉及哪些共享资源？谁可能在竞争？
   - **反馈/放大**：有没有正反馈回路？（重试、告警触发、自动扩缩容）
   - **数据流**：（如涉及数据问题）数据从哪来？经过什么转换？
4. **标注世界可达性**：每个识别出的世界，标注 ✅⚠️❌
5. **向用户确认**：世界模型是否完整？有没有遗漏的中间层或变更事件？
6. 检查 project-cognition 中是否有相关历史经验

---

### Phase 1: 初始观测 (Observe)

**调用 Evidence Builder** 在可见世界中采集证据。

**关键**: 不仅采集"有什么"，还要主动检查"没有什么"（缺失事实）。
不可达的世界记录为缺失事实，影响置信度上限。

---

### Phase 2: 假设生成 (Hypothesize)

基于当前证据包和多视角世界模型，生成候选假设。

**假设从视角产出生成，不是从抽象分类生成：**

- 执行路径的每个关键节点 → 对应假设
- 每个相关变更事件 → 对应假设
- 每个资源竞争点 → 对应假设
- 每个反馈回路 → 对应假设

**假设生成规则：**

- 不能只有一个假设——至少要有竞争假设来保持诚实
- 假设应覆盖多个视角（不要全在同一个视角里）
- 每个假设必须能被证伪
- **为盲区世界也生成假设**——即使无法验证，它们的存在提醒你置信度有天花板

#### 假设生成检查点

生成假设后，快速自检：

- [ ] 执行路径上的关键节点都覆盖了吗？
- [ ] 考虑了"什么变了"的可能性吗？
- [ ] 考虑了资源竞争的可能性吗？（如果症状涉及资源）
- [ ] 考虑了反馈放大的可能性吗？（如果症状涉及流量/级联）
- [ ] 为盲区世界生成了假设吗？

**任何一项是"否"且相关，必须补充假设。**

---

### Phase 3: 假设验证 (Validate)

**调用 Hypothesis Validator** 做四层校验（时间 → 量级 → 范围 → 反事实）。

```bash
python3 hypothesis_validator/scripts/validate_hypothesis.py --input evidence.json --pretty
```

然后由 LLM 补充反事实校验。

#### 验证上下文一致性原则

**任何验证结果的可靠性，取决于验证上下文与故障上下文的一致程度。**

上下文维度：
- **位置**：在哪里执行的？（本地 vs Pod vs 跳板机 vs 生产环境）
- **时间**：什么时候的数据？（故障时 vs 现在）
- **输入**：用什么数据？（故障数据 vs 构造数据）
- **配置**：用什么版本？（故障时的配置 vs 当前配置）

**上下文不一致的验证，不能确定性地推翻假设**，只能作为参考降低可能性。

> 例：本地 curl 能通 ≠ Pod 里能通。如果故障发生在 Pod，必须在 Pod 里验证。

#### 验证检查点

用验证结果做决策前，快速自检：

- [ ] 验证的执行位置与故障位置一致吗？
- [ ] 验证的时间窗口与故障时间相关吗？
- [ ] 如果上下文不一致，我是否降级了这个验证的可靠性？

**如果用不一致上下文的验证推翻了假设，这个推翻可能是错的。**

---

### Phase 4: 决策 (Decide)

- **有假设被 promote** → 收敛，Phase 5
- **证据不足 (insufficient_evidence)** → 扩展世界，回 Phase 1
- **有 retain 但无法区分** → 扩展世界
- **全部 reject** → 生成新假设或带不确定性收敛
- **到达最大轮次** → 带不确定性收敛

#### 扩展世界

按**判别性信息增益**排序（见 investigation-loop-protocol.md），
选最能区分竞争假设的世界扩展。新证据增量追加到 Evidence Pack。

#### 当世界进不去时

**这是最关键的分支。不要静默跳过，不要硬在已有世界里凑答案。**

向用户坦诚：

> "我已排查了 [code/log/trace] 世界的证据，但不足以解释症状。
> 我怀疑问题可能在 [deploy/network/infra] 层面，但我无法进入这些世界。
> 你能帮我确认：[具体问题]？"

**一个诚实的"我不知道"胜过一个自信的错误结论。**

---

### Phase 5: 收敛与报告 (Converge)

报告是给人看的，用 markdown。结构：

1. **一句话结论**（或一句话标注不确定性边界）
2. **关键证据链**——按时间排列的因果链
3. **被推翻的假设**——推翻理由是调查严谨性的证明
4. **剩余不确定性**——哪些世界没进入？哪些假设无法验证？
5. **调查过程摘要**

---

### Phase 6: 处置 (Resolve)

处置方案分四级: 🔴立即止血 → 🟡短期修复 → 🔵长期预防 → 📊持续监控

执行原则——**按副作用分级**:

| 副作用 | 处理方式 |
|--------|---------|
| 无（只读/分析/创建分支） | 直接执行 |
| 低（本地配置/新分支提交） | 直接执行 |
| 中（扩缩容/限流） | 征询用户后执行 |
| 高（回滚/重启/生产变更） | 征询用户后执行，或只给方案 |

---

## 用户交互

调查不是闭门造车。**但只在真正需要时才问用户**——能自己查到的不要问。

核心交互时机:

| 时机 | 做什么 |
|------|--------|
| Phase 0 | 确认症状提取、补充上下文 |
| 可见世界的假设解释不了症状时 | 坦诚盲区，请求帮助进入新世界 |
| 处置方案生成后 | 确认方案、授权有副作用的操作 |

## 经典案例族：从故障族和操作边界指明方向

以下案例展示如何用故障族和操作边界指明正确方向。**目标不是精确定位，而是确保方向对。**

### 案例 1：级联雪崩（Spotify 2013）

**症状**：微服务 A 调用 B，B 调用 C。C 开始变慢，然后整个系统雪崩，所有服务不可用。

**故障族**：级联/放大 + 资源/竞争

**操作边界**：
```
A → B → C (变慢)
    ↺ 重试     → 请求量放大
↺ 重试         → 进一步放大
```

**正确的假设方向**：
- H1: C 本身变慢（依赖/第三方）
- H2: 重试策略放大了问题（反馈/放大）← 关键方向
- H3: 没有熔断/降级保护（反馈/放大）
- H4: 线程池/连接池耗尽（资源/竞争）

**错误方向**：只盯着 C 为什么慢，忽略放大机制。

**正确的方向输出**：
> "这是**级联/放大族**问题。C 变慢只是触发点，真正的问题是重试策略导致请求量指数放大。应该检查重试配置、熔断机制、以及各层的资源池状态。"

---

### 案例 2：配置变更（Cloudflare 2022 BGP）

**症状**：多个数据中心同时不可用，持续约 30 分钟。

**故障族**：配置/变更

**操作边界**：
```
运维变更 → BGP 配置生效 → 路由失效 → 流量无法到达
          ^
          最远确认（有变更记录）
```

**正确的假设方向**：
- H1: 最近有配置/发版变更（时间/变更）← 关键方向
- H2: 硬件故障（但多数据中心同时？概率极低）
- H3: 外部攻击（DDoS）

**错误方向**：检查各个数据中心的服务状态，忽略"多数据中心同时"这个信号。

**正确的方向输出**：
> "多数据中心同时故障，极可能是**配置/变更族**问题。应该检查最近的全局配置变更、网络配置变更、DNS 变更。"

---

### 案例 3：资源竞争（PostgreSQL TXID Wraparound）

**症状**：数据库突然变成只读，所有写入失败。无任何发版或配置变更。

**故障族**：资源/竞争 + 数据/状态

**操作边界**：
```
应用写入 → DB 接收 → 事务分配 TXID → 写入失败
                      ^
                      第一处未确认（TXID 分配发生了什么？）
```

**正确的假设方向**：
- H1: 磁盘满（资源/竞争）
- H2: 内部资源耗尽（TXID、连接数等）（资源/竞争）← 关键方向
- H3: 数据库 bug（依赖/第三方）
- H4: 权限变更（配置/变更）

**错误方向**：检查应用代码、检查网络连接、检查最近发版。

**正确的方向输出**：
> "数据库突然只读，无外部变更，应该是**资源/竞争族**问题。检查数据库内部资源：磁盘空间、TXID 使用率、连接数、锁等待。"

---

### 案例 4：第三方库 Bug（内存泄漏）

**症状**：服务 OOM 重启，重启后短暂正常，然后又 OOM。

**故障族**：依赖/第三方 + 资源/竞争

**操作边界**：
```
服务启动 → 正常运行 → 内存持续上升 → OOM
                      ^
                      第一处未确认（内存为什么持续上升？）
```

**正确的假设方向**：
- H1: 代码内存泄漏（代码问题）
- H2: 第三方库内存泄漏（依赖/第三方）← 可能方向
- H3: 最近升级了某个依赖（时间/变更 + 依赖/第三方）
- H4: 缓存配置不当（配置/变更）

**错误方向**：只看自己的代码，忽略依赖库。

**正确的方向输出**：
> "持续内存上升直到 OOM，可能是**资源/竞争族**（内存泄漏）。应该检查：最近依赖升级、代码变更、缓存配置。用 profiler 定位泄漏点，注意可能在第三方库里。"

---

### 案例 5：数据污染（OpenAI 2023 Redis 队列）

**症状**：部分用户看到了其他用户的对话历史。

**故障族**：数据/状态

**操作边界**：
```
用户请求 → 队列读取 → 返回数据
           ^
           第一处未确认（队列里的数据是谁的？）
```

**正确的假设方向**：
- H1: 缓存 key 冲突（数据/状态）
- H2: 队列顺序错乱（数据/状态）← 关键方向
- H3: 并发控制问题（资源/竞争）
- H4: 最近缓存/队列配置变更（配置/变更）

**错误方向**：检查用户认证、权限系统。

**正确的方向输出**：
> "用户看到他人数据，这是**数据/状态族**问题。应该检查：缓存 key 生成逻辑、队列顺序、并发下的数据隔离。"

---

### 从案例中学到的

1. **故障族决定关键视角**：
   - 级联雪崩 → 反馈/放大视角
   - 配置故障 → 时间/变更视角  
   - 资源耗尽 → 资源/竞争视角
   - 数据问题 → 数据流视角

2. **操作边界决定下一步**：
   - 边界推进不了 → 向用户求助进入那个世界
   - 边界已经推进 → 继续往下一跳看

3. **"所有假设都在同一个族"是危险信号**：
   - 应该强制从另一个族生成假设
   - 多数据中心同时故障，还猜"某个服务 bug"是错的

## 调查过程沉淀（Level 2）

**Level 2 深度调查过程中，将关键发现实时沉淀到 `.claude/shadow-claw/` 目录：**

### 调查进度文件

```
.claude/shadow-claw/
  └── investigation-{case_id}.md
```

**内容结构**：
```markdown
# 调查进度：{症状简述}

## 元信息
- Case ID: {trace_id 或自动生成}
- 开始时间: {ISO8601}
- 当前轮次: {N}/5
- 状态: {investigating | converged | blocked}

## 故障族与操作边界
- 故障族: {identified}
- 操作边界: {最远确认} → {第一处未确认}

## 假设状态
| ID | 假设 | 状态 | 理由 |
|----|------|------|------|
| H1 | ... | retain/reject/insufficient | ... |

## 关键证据
1. {证据1}
2. {证据2}

## 盲区与待验证
- 进不去的世界: [...]
- 需要用户帮助: [...]

## 下一步
- [ ] {下一步行动}
```

**沉淀时机**：
- Phase 0 完成后（故障族、操作边界确定）
- 每轮假设验证后（假设状态更新）
- 发现盲区时
- 向用户求助前

**价值**：
- 用户可以随时查看调查进度
- 调查中断后可以继续
- 其他 Agent 可以接手
- 为后续类似问题提供参考

## 经验沉淀

调查结束后，有价值的模式沉淀到 project-cognition:

- 排障模式 → `project_cognition/runbooks/`
- 系统知识 → `project_cognition/domain-knowledge/`
- 原则/禁区 → `project_cognition/core-manifest/`

**Level 2 调查完成后，如果发现了有价值的模式**：

```markdown
# 沉淀示例：跨服务超时排查

## 症状模式
- A 服务调 B 服务超时
- 错误信息包含 "context deadline exceeded" 或 "Client.Timeout"

## 排查路径
1. 先确认操作边界：请求从哪到哪，卡在哪一跳
2. 区分：DNS 问题 / IP 可达性 / 上游服务慢
3. 从故障位置验证（Pod 内部），不要在本地验证

## 常见陷阱
- 本地 curl 能通不代表 Pod 里能通（网络拓扑不同）
- "上游慢"是最常见猜测，但要先验证网络层
```

## 反模式自检

**每轮结束时，花 5 秒自检以下常见错误：**

### Level 相关反模式

| 反模式 | 症状 | 修正 |
|--------|------|------|
| **简单问题过度调查** | 明显的语法错误/配置缺失却启动 Level 2 | 降级到 Level 0/1，快速解决 |
| **复杂问题不升级** | Level 1 假设被推翻，还在硬凑解释 | 立即升级到 Level 2 |
| **跳过升级检查** | Level 1 给结论前没有做升级检查 | 回退，执行升级检查清单 |

### 致命反模式（Level 2 必须检查）

| 反模式 | 症状 | 修正 |
|--------|------|------|
| **跳过世界模型** | 看到症状直接猜原因，没有构建多视角理解 | 回退到 Phase 0，从五个视角过一遍 |
| **单视角困住** | 假设全在同一个视角里（如全是代码问题） | 强制从其他视角生成至少一个假设 |
| **上下文不一致验证** | 用本地验证推翻 Pod 里的故障 | 降级验证可靠性，不能确定性推翻 |
| **第一轮就收敛** | 第一个看起来合理的解释就接受了 | 至少再扩展一个世界，找竞争证据 |
| **盲区静默跳过** | 某个世界进不去，但没有记录也没有告诉用户 | 显式记录盲区，生成对应假设，坦诚告知 |

### 常见偏差（需要警惕）

| 偏差 | 表现 | 对策 |
|------|------|------|
| **熟悉度偏差** | 只在熟悉的世界里找（如只看代码） | 问自己：有没有我不熟悉但可能相关的世界？ |
| **确认偏差** | 只找支持当前假设的证据 | 主动寻找反驳当前假设的证据 |
| **锚定效应** | 第一个假设影响后续所有判断 | 把假设打乱顺序重新评估 |
| **过度自信** | 盲区很多但给出高置信度 | 置信度 ≤ 世界覆盖率 |

### 方向偏离检测

**如果发现以下情况，可能已经偏离了正确方向：**

| 信号 | 可能的问题 | 修正 |
|------|-----------|------|
| 两轮假设都在同一个故障族 | 遗漏了其他故障族 | 强制从另一个故障族生成假设 |
| 操作边界一直没有推进 | 卡在无法验证的边界 | 向用户求助进入那个世界 |
| 所有假设都指向"应用层" | 忽略了更底层的物理链路 | 检查网络、存储、基础设施层 |
| 没有假设涉及"最近变了什么" | 忽略了时间/变更视角 | 查询最近的发版、配置、依赖变更 |
| 验证总在"可以验证"的世界里打转 | 避开了真正需要验证但进不去的世界 | 坦诚盲区，向用户求助 |

**方向偏离比错误假设更危险** — 错误假设可以被推翻，但如果方向本身偏了，所有假设都在错误的空间里。

**如果发现自己触发了任何反模式或方向偏离，必须回退并修正，不是继续往前走。**

---

## 安全约束

1. **不在证据不足时给唯一结论** — 宁可说"我还不确定，我的盲区在 X"
2. **不跳过世界模型直接猜答案** — 必须从多视角构建理解
3. **不跳过假设直接给答案** — 必须有竞争假设
4. **不无限循环** — 最多 5 轮
5. **不忽略 insufficient_evidence** — 必须扩展世界或承认无知
6. **不在调查收敛前改代码** — 铁律
7. **不在盲区里给高置信度结论** — 置信度受世界覆盖率约束
8. **不用不一致上下文的验证确定性推翻假设** — 只能降级可能性
