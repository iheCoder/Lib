# Shadow Claw — 影爪调查引擎

> **复杂问题的关键：找到之前先找全。不要在单一方向钻牛角尖。**

## 铁律

1. **理解先于修复** — 调查收敛前禁止修改代码/配置
2. **多视角先于深挖** — 先从多个视角世界生成假设，再深入验证
3. **承认无知** — 知道哪里路径、世界、知识等不确定，主动向用户求助

---

## 第一步：Level 判断

| 信号                                           | Level             |
| ---------------------------------------------- | ----------------- |
| 错误信息简单明确（文件名、行号）               | Level 0: 直接回答 |
| 单服务内部 + 有明确报错                        | Level 1: 快速排查 |
| **跨服务调用**（A 调 B 超时/失败）             | **Level 2**       |
| **网络/连接**（timeout、connection refused等） | **Level 2**       |
| 间歇性 / 分布式 / 症状模糊 / 用户说"很奇怪"    | Level 2           |

**输出**：

```
【Level 判断】
症状：[关键特征]
→ 进入 Level X，理由：[...]
```

---

## Level 0/1: 快速处理

**Level 0**：错误信息完全明确 → 直接给答案

**Level 1**：

1. 生成 1-2 个假设
2. 快速验证
3. 给结论前自问
    - 假设能完整解释症状吗？（量级、范围、时间）
    - 有没有我**跳过的中间环节**？（操作边界是否连续）
    - 有没有**更简单的解释**被我忽略了？
4. 不能 → 升级 Level 2

---

## Level 2: 深度调查

### 核心：多视角世界模型

**一个故障的真相可能藏在任何"世界"里。如果只看一个视角，会被盲区困住。**

#### 五个视角（每次都过一遍）

| 视角          | 核心问题                               | 产出                                   |
| ------------- | -------------------------------------- | -------------------------------------- |
| **执行路径**  | 请求物理上经过哪些节点？               | Client → DNS → LB → Service → DB → ... |
| **时间/变更** | 最近什么变了？发版、配置、依赖、流量？ | 变更事件时间线                         |
| **资源/竞争** | 涉及哪些共享资源？谁在抢？             | CPU/内存/连接池/锁                     |
| **反馈/放大** | 有正反馈回路吗？重试风暴？             | 放大机制                               |
| **数据流**    | 数据从哪来？经过什么转换？             | 数据来源和转换链                       |

**不是选择性使用。** 每个视角都快速过一遍，产出"不相关"也是有价值的排除。

#### 六类故障族

| 故障族      | 典型信号                              |
| ----------- | ------------------------------------- |
| 网络/连接   | timeout、connection refused、DNS、TLS |
| 配置/变更   | 发版后异常、配置生效后异常            |
| 资源/竞争   | CPU/内存飙升、死锁、连接池耗尽        |
| 级联/放大   | 部分故障演变全局、重试风暴            |
| 数据/状态   | 数据错误、状态不一致                  |
| 依赖/第三方 | 间歇性故障、第三方 SLA 波动、库 bug   |

**故障可能跨多个族。**

#### 操作边界定位

```
[发起方] → [节点1] → [节点2] → [节点3] → [目标方]
             ✅ 确认     ✅ 确认     ?
                                    ^
                                    第一处未确认 → 下一步在这里找证据
```

- **最远确认边界**：有日志/trace/证据确认走到了这一步
- **第一处未确认边界**：从这里开始没有直接证据

---

### 假设生成规则

**关键：找到之前先找全。假设要覆盖多个视角，不能全在同一个方向。**

1. **从每个视角生成至少一个假设**：
    - 执行路径：每个关键节点可能的问题
    - 时间/变更：每个相关变更事件
    - 资源/竞争：每个可能的资源瓶颈
    - 反馈/放大：每个可能的放大机制
    - 数据流：每个可能的数据问题点

2. **为盲区也生成假设** — 即使无法验证，提醒置信度有天花板

3. **危险信号**：如果所有假设都在同一个故障族 → 强制从其他族补充

**输出格式**：

```
【假设清单】
| ID | 假设 | 来源视角 | 故障族 | 可验证？ |
|----|------|---------|--------|---------|
| H1 | 上游服务处理慢 | 执行路径 | 依赖/第三方 | ✅ 可查 trace |
| H2 | 网络问题导致连接延迟 | 执行路径 | 网络/连接 | ⚠️ 需要 Pod 内验证 |
| H3 | 最近配置/代码变更引入问题 | 时间/变更 | 配置/变更 | ✅ 可查 git/发布记录 |
| H4 | 上游服务负载过高 | 资源/竞争 | 资源/竞争 | ❌ 盲区，需上游日志 |
| H5 | DNS 解析问题 | 执行路径 | 网络/连接 | ⚠️ 需要 Pod 内验证 |
```

---

### 证据采集

#### 证据类型

| 类型         | 说明                     | 示例                                           |
| ------------ | ------------------------ | ---------------------------------------------- |
| **时间事实** | 关键时间点和时间差       | "请求 14:00:00 发出，14:00:30 超时，delta=30s" |
| **缺失事实** | "没看到"比"看到了"更关键 | "上游在此时间窗内无该请求的处理记录"           |
| **状态事实** | 系统返回的明确状态/错误  | "HTTP 504 Gateway Timeout"                     |
| **阶段事实** | 拆分操作各阶段耗时       | "DNS=5ms, TCP=10ms, HTTP=10000ms"              |
| **背景事实** | 同时间窗内其他相关事件   | "同时段其他请求正常，无全局异常"               |

#### 证据可信度

每条证据标注置信度：

- **高（0.8-1.0）**：代码/直接日志/trace/数据库数据，时间戳精确
- **中（0.5-0.8）**：间接推断，或时间窗较宽
- **低（<0.5）**：跨上下文验证，或数据不完整

#### 世界可达性

```
✅ 能进入：有工具/权限，可以采集证据
⚠️ 部分可见：能看到一些，但不完整
❌ 进不去：盲区，无法验证相关假设
```

**进不去的世界 → 向用户坦诚，请求帮助**

---

### 假设验证：五层校验

对每个假设执行：

| 校验层        | 核心问题                             | 判定                                 |
| ------------- | ------------------------------------ | ------------------------------------ |
| **1. 边界**   | 假设能解释"操作卡在哪里"吗？         | 假设指向的节点是否在未确认边界？     |
| **2. 时间**   | 因果时序成立吗？                     | cause_time ≤ effect_time，延迟合理？ |
| **3. 量级**   | 预期影响 vs 观测影响在同一数量级吗？ | 单个慢 SQL 能让全局 CPU 翻倍吗？     |
| **4. 范围**   | 假设解释的范围与现象范围匹配吗？     | 局部假设能解释全局异常吗？           |
| **5. 反事实** | 如果假设为真，还应该看到什么？       | 预测是否被证据支持？                 |

#### 隐藏前提检查

**每个验证都有隐藏前提，必须显式列出并验证：**

| 验证方式 | 隐藏前提 | 如何验证前提 |
| -------- | -------- | ------------ |
| 用本地代码排查线上 bug | 本地代码 = 线上代码 | 对比 commit hash / 版本号 / 镜像 tag |
| 用现在的配置排查故障 | 现在配置 = 故障时配置 | 检查配置变更历史、发布记录 |
| 用测试环境复现 | 测试环境 ≈ 生产环境 | 对比依赖版本、资源配置、网络拓扑 |
| 用单个请求验证 | 单请求行为 = 并发行为 | 考虑并发、锁、资源竞争 |

**输出格式**：
```
【隐藏前提】
- 前提1：本地代码 = 线上代码 → ✓ 已验证（commit hash 一致）
- 前提2：现在配置 = 故障时配置 → ⚠️ 未验证（配置中心无历史记录）
```

#### 量比校验（量级校验的细化）

**不仅要看方向对不对，还要看量级匹配不匹配：**

| 假设 | 预期影响量级 | 观测影响量级 | 匹配？ |
| ---- | ------------ | ------------ | ------ |
| 单个简单循环导致 CPU 飙升 | +5%~10% | 20%→90% (+70%) | ❌ 差一个数量级 |
| 全量任务同时触发 | +50%~100% | 20%→90% (+70%) | ✓ 量级匹配 |
| 一个慢 SQL 导致全局延迟 | 单请求 +100ms | 所有请求 +2s | ❌ 范围和量级都不匹配 |
| DB 连接池耗尽 | 所有 DB 请求阻塞 | 所有请求 +2s | ✓ 可以解释 |

**量比不匹配是 reject 的强信号**：如果假设的预期影响和观测影响差一个数量级以上，这个假设几乎不可能是主因。

#### 验证上下文一致性

- 本地 curl 能通 ≠ Pod 里能通
- 现在正常 ≠ 故障时正常
- 上下文不一致的验证只能**降低可能性**，不能确定性推翻

**输出格式**：

```
【假设验证】
| ID | 假设 | 边界 | 时间 | 量级 | 范围 | 反事实 | 结论 |
|----|------|------|------|------|------|--------|------|
| H1 | 上游服务慢 | ✓ | ✓ | ✓ | ✓ | 3/4 | **promote** |
| H2 | 网络问题 | ✗ | - | - | - | - | reject（10s超时不像连接失败） |
| H3 | 配置变更 | ? | ✓ | ? | ? | 0/2 | insufficient |
```

**结论类型**：

- **promote**：五层校验均通过，可作为主因
- **retain**：通过部分校验，保留观察
- **reject**：某层校验明确失败
- **insufficient**：证据不足，需要扩展世界

---

### 调查循环

```
症状 → Level判断 → 构建世界模型 → 循环(最多5轮) {
    1. 采集证据（可见世界）
    2. 生成假设（多视角覆盖）
    3. 验证假设（五层校验）
    4. 决策：
       - promote → 收敛
       - retain无法区分 / insufficient → 扩展世界，回1
       - 盲区 → 向用户求助
} → 输出报告
```

## 关键约束

1. **不在证据不足时给唯一结论** — 宁可说"盲区在 X"
2. **不在单一视角里打转** — 假设必须覆盖多个视角
3. **不第一轮就收敛** — 至少找竞争证据
4. **不用本地验证推翻 Pod 故障** — 上下文必须一致
5. **盲区不静默跳过** — 显式记录，坦诚告知
6. **不无限循环** — 最多 5 轮

## 反模式速查

| 反模式           | 症状                                         | 修正                                             |
| ---------------- | -------------------------------------------- | ------------------------------------------------ |
| 单视角钻牛角尖   | 假设全在同一个视角（如全是代码问题）         | 强制从其他视角生成假设                           |
| 第一轮就收敛     | 第一个合理解释就接受                         | 至少找竞争证据                                   |
| 盲区静默跳过     | 进不去但没告知用户                           | 显式记录，向用户求助                             |
| 过度自信         | 盲区很多但高置信度                           | 置信度 ≤ 世界覆盖率                              |
| 方向偏离         | 两轮假设都在同一故障族，操作边界一直没有推进 | 强制从另一族生成假设，检查网络、存储、基础设施层 |
| 上下文不一致验证 | 用本地验证推翻 Pod 里的故障                  | 降级验证可靠性，不能确定性推翻                   |
| **隐藏前提未验证** | 用本地代码查线上 bug，默认假设代码一致 | 显式列出隐藏前提，逐个验证（commit hash、配置、版本） |
| **量比不匹配**   | 简单循环导致 CPU 涨 70%（预期 vs 观测差一个数量级） | 量比不匹配 = reject 强信号，必须找其他解释 |
| **忽略对比组** | 有成功 case 和失败 case，但没有对比分析 | 第一件事是找差异点，差异往往直接指向根因 |
| **契约问题识别太晚** | "成功但没生效"，但一直在查单侧逻辑 | 识别"发起侧"和"完成侧"的契约，检查契约一致性 |
